# `cartograph generate` — Design Spec

## Goal

Read the indexed codebase data from Postgres and produce a set of static markdown files that AI coding tools consume on session start. The AI begins every session already knowing the project's architecture, module layout, dependency graph, and coding conventions — zero tokens wasted on orientation.

## Output Structure

All generated files live under `<repo-path>/.cartograph/`. The tool never touches files outside this directory.

```
.cartograph/
├── CLAUDE.md          — lean root overview (~3K tokens)
├── modules.md         — what exists where, grouped by area (~5K tokens)
├── dependencies.md    — directed module dependency graph (~3K tokens)
└── conventions.md     — detected structural patterns and naming (~2K tokens)
```

Every file starts with:
```markdown
<!-- Generated by Cartograph. Do not edit manually. -->
```

After generation, the CLI prints:
```
✅ Generated .cartograph/CLAUDE.md + 3 topic files
   Add this to your root CLAUDE.md:
   Also read: .cartograph/CLAUDE.md
```

## Architecture

### File Structure

This is a new `src/output/` module — create the directory.

```
src/output/
├── generate-pipeline.ts     — coordinator: queries DB, calls generators, writes files
├── root-generator.ts        — produces CLAUDE.md
├── modules-generator.ts     — produces modules.md
├── deps-generator.ts        — produces dependencies.md
└── conventions-generator.ts — produces conventions.md
```

### Data Flow

1. **Coordinator** (`generate-pipeline.ts`) queries the DB once upfront and maps results into clean interfaces.
2. **Generators** are pure functions: `(data: SomeInput) => string`. They receive plain objects, return markdown strings. No DB access, no side effects.
3. **Coordinator** writes each returned string to its file under `.cartograph/`.

This means generators are trivially unit-testable with fake data — no DB setup required.

### DB Queries

The coordinator resolves `<repo-path>` to an absolute path (via `path.resolve()`) before DB lookup, matching the normalization applied during `index` (which also calls `resolve(repoPath)`). This ensures `cartograph generate ./my-project` and `cartograph generate /abs/path/to/my-project` both find the same repo.

Requires adding `findByPath(path: string): Promise<RepoRecord | null>` to `RepoRepository` — a read-only lookup that returns null instead of creating a new repo.

The coordinator runs three query groups:

1. **Files + symbols** — all files with their symbols joined, for grouping by directory. Feeds root + modules generators.
2. **Symbol references** — all resolved references with source/target qualified names and reference kinds. Feeds deps + conventions generators.
3. **Aggregate stats** — counts by kind (files, symbols, references), top-level directory breakdown. Feeds root generator.

### CLI Command

```
cartograph generate <repo-path> [--output-dir <path>]
```

- `<repo-path>` — path to the indexed repository (also used for config loading, matching the `index` command pattern)
- `--output-dir` — defaults to `<repo-path>/.cartograph/`. Override to write output elsewhere.

Follows the existing CLI pattern: `loadConfig()` → `createPool()` → work → `pool.end()`.

## File Specifications

### `.cartograph/CLAUDE.md` (~3K tokens)

The entry point. An AI reads this first and knows enough to navigate.

**Sections:**
- **Project Overview** — language, file count, symbol count (one line)
- **Architecture** — 2-3 sentences derived from top-level directory structure and dominant patterns (e.g., "MVC with service layer, repository pattern for data access")
- **Directory Map** — each top-level directory with symbol count and one-line description derived from the dominant class kinds there (e.g., `app/Models/ — 45 classes (Eloquent models, core domain)`)
- **Context Files** — links to the three topic files with one-line descriptions

**Architecture description heuristic:** derived from directory names + symbol kinds. If there's a `Services/` + `Repositories/` + `Models/`, it's "service-repository pattern". If there's `Controllers/` + `Views/`, it's MVC. This is pattern-matched, not LLM-generated.

### `.cartograph/modules.md` (~5K tokens)

Grouped class inventory with key relationships.

**Structure:**
- One section per top-level namespace/directory
- Within each section: classes listed with kind, line count, and key relationships (implements X, extends Y, uses trait Z)
- Classes ordered by reference count (most-connected first = most important)
- For large areas (50+ classes): show top 10 by reference count, summarize the rest as "... and N more"

**Example:**
```markdown
## Services (38 classes)

| Class | Implements | Key Dependencies |
|-------|-----------|-----------------|
| UserService | UserServiceInterface | UserRepository, User |
| OrderService | OrderServiceInterface | OrderRepository, NotificationService |
| ... | ... | ... |

... and 28 more classes
```

### `.cartograph/dependencies.md` (~3K tokens)

Directed module-level dependency graph. Shows which top-level namespaces depend on which others, with reference counts and directionality.

**Structure:**
- **Module Dependencies** — each module pair with direction and count, sorted by reference count descending
- **Dependency Violations** — cases where lower layers reference upper layers (e.g., Models → Services), flagged as potential architectural violations
- **External Dependencies** — top frameworks/libraries referenced but not in the codebase (e.g., Illuminate, Symfony), with reference counts. Group by first namespace segment of `target_qualified_name` (e.g., `Illuminate\Support\Facades\DB` and `Illuminate\Database\Eloquent\Model` both roll up under "Illuminate")

**Directionality is critical.** Not "Services connects to Models" but:
```
Services → Models (142 refs)
Models → Services (0 refs)  ← clean layering
```

This tells the AI about the architectural layering and helps it avoid violations.

**Layer detection heuristic:** infer expected layering from conventional directory names:
- Controllers > Services > Repositories > Models (typical Laravel/MVC)
- Any reference going "upward" in this hierarchy is flagged as a potential violation

### `.cartograph/conventions.md` (~2K tokens)

Structural patterns detected from the index. No LLM — pure stats from what's actually in the code.

**Sections:**
- **Structural Patterns** — e.g., "94% of Services implement a matching interface in Contracts/", "All Repositories extend BaseRepository"
- **Naming Conventions** — class naming (PascalCase %), method naming (camelCase %), namespace-to-directory mapping
- **Composition Patterns** — trait usage distribution (which areas use traits, which don't), interface adoption rate per module, inheritance depth stats

**Detection approach:**
- Query symbol kinds + reference kinds per module
- Compare naming against standard patterns (regex-based)
- Compute percentages for structural patterns (e.g., "X% of classes in Y/ implement an interface")
- Flag any outliers as "exceptions" (e.g., "3 services don't implement an interface: FooService, BarService, BazService")

## Token Budget Enforcement

Each generator has a soft token budget. When output would exceed the budget:
- **Truncate with summary** — "... and N more" rather than listing everything
- **Prioritize by reference count** — most-connected symbols appear first since they're most likely to be relevant
- **Token estimation** — approximate 1 token ≈ 4 characters for English/code markdown. No need for a tokenizer.

## Error Handling

- **Empty index** — print "No indexed data found for this repository. Run `cartograph index` first." and exit with code 1.
- **Existing `.cartograph/`** — overwrite all files silently. Generate is idempotent.
- **Repo not found in DB** — print "Repository not indexed: <path>. Run `cartograph index <path>` first." and exit with code 1.
- **Write failures** — throw `GenerateError` (new error type in `src/errors.ts`, following existing `CartographError` pattern with code `'GENERATE_ERROR'`).

## Commit Policy

`.cartograph/` is intended to be committed to the repository. AI tools read these files on session start. Do not add `.cartograph/` to `.gitignore`.

## Testing Strategy

- **Unit tests** for each generator — pass fake data, assert output contains expected sections, stats, and formatting. Test truncation behavior with large inputs.
- **Integration test** — index the laravel-sample fixture, run generate pipeline, verify:
  - Four files created in `.cartograph/`
  - Each has the generated header
  - Content is non-empty and structurally correct
  - Re-running produces identical output (idempotent)
- **No snapshot tests** — output format will evolve; structural assertions are more maintainable.

## Scope Boundaries

**In scope for v1:**
- Four files: CLAUDE.md, modules.md, dependencies.md, conventions.md
- Pure stats-based conventions (no LLM)
- Module-level dependency graph (namespace/directory level, not class-level)
- Token budget enforcement via truncation

**Out of scope (v2+):**
- Per-subdirectory CLAUDE.md files
- Flow docs (`.cartograph/flows/`)
- LLM-based pattern summarization for conventions
- Constructor injection detection (requires signature string parsing — v1 conventions stick to what's directly queryable from symbol_references)
- Class-level dependency graphs
- Custom output templates
